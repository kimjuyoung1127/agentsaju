# 사주 라이프 시뮬레이터 PRD

Project: `Saju Life Simulator` (가칭)  
Version: `1.0.0`  
Date: `2026-03-17 (KST)`  
Context: 이 문서는 `researchpromt.md`, `result1.md`, `result2.md`(=`result3.md`), `result4.md`를 통합해 만든 실제 제품 요구사항 문서다. MiroFish는 코드 재사용 대상이 아니라 "패턴 참고용"이며, 본 서비스는 `클린룸 재구현`을 전제로 한다.

## 0. 고정 의사결정
- 제품 포지셔닝은 "텍스트 운세 앱"이 아니라 `내 AI 아바타가 1년을 먼저 살아보는 사주 기반 라이프 시뮬레이터`다.
- 사주 계산의 코어는 `결정론 엔진`이 담당하고, LLM은 `설명/요약/대안 제시`에만 사용한다.
- 시간축은 `월 단위 기본 + 주 단위 확대(유료)`를 기본 모델로 삼는다.
- 핵심 도메인은 `재물`, `커리어`, `관계`, `건강` 4개이며 모든 화면과 데이터 구조는 이 4축을 중심으로 설계한다.
- 결제는 "가장 중요하지만 설명 가능성이 높은 사건"에서 유도한다. 즉, `importance`가 높고 `confidence`가 충분하며 `avoidance_plan`이 존재하는 이벤트가 페이월 포인트다.
- 개인정보와 법적 리스크가 크므로 `출생 정보 최소 수집`, `암호화 저장`, `삭제권`, `엔터테인먼트 목적 면책`, `공인/유명인 악용 방지`를 MVP부터 포함한다.
- AGPL 비수용 전제이므로 MiroFish의 코드/프롬프트/표현/파일 구조는 복제하지 않는다. 참고 가능한 것은 `비동기 작업`, `로그 구조`, `리포트 섹션화`, `실시간 진행 표시` 같은 패턴뿐이다.

## 1. 문제 정의
기존 사주/운세 서비스의 주된 한계는 다음과 같다.

- 결과가 정적인 텍스트라서 "왜 이런 결론이 나왔는지" 설명이 약하다.
- 월운/대운이 있어도 실제 의사결정 맥락(이직, 연애, 투자, 동업)까지 연결되지 않는다.
- 결과가 단발성 소비로 끝나며, 타임라인/이벤트/백테스트/회피 루트처럼 반복 방문을 만들 장치가 약하다.
- 결제 포인트가 랜덤하거나 조급해 보여 신뢰를 깎기 쉽다.
- LLM만으로 운세를 만들면 재현성과 설명 가능성이 붕괴한다.

## 2. 제품 비전
한 줄 가치 제안:

> 사용자의 사주와 현실 목표를 입력하면, 나를 닮은 AI 아바타가 1년을 먼저 살아보며 월별 변곡점, 충돌, 기회, 회피 루트를 사건 카드와 리포트로 보여주는 서비스

제품이 반드시 주는 경험:

- 사용자는 `연간 흐름`을 한눈에 본다.
- 사용자는 `특정 월`을 눌러 더 자세한 주간 리스크/기회를 탐색한다.
- 사용자는 단순 길흉이 아니라 `무엇이 원인이었는지`, `어떤 선택지가 있었는지`, `어떻게 피할 수 있는지`를 이해한다.
- 사용자는 과거 연도 백테스트를 통해 "이 엔진이 적어도 내 체감과 얼마나 맞는지" 검증할 수 있다.

## 4. 목표와 비목표

### 4.1 제품 목표
- 가입 후 `차트 컴파일 성공률 98% 이상`
- 연간 기본 시뮬레이션(`12개월`)의 `완료 시간 60초 이내`
- 월 확대(주 단위) 요청의 `첫 응답 스트리밍 시작 5초 이내`
- 무료 사용자의 `클라이맥스 카드 노출 대비 결제 페이지 진입률 20% 이상`
- 결제 페이지 진입자의 `결제 전환율 3~5% 이상`
- 백테스트 참여자의 `재방문율`을 일반 사용자보다 높게 만든다.
- "왜 이 결과가 나왔는가"를 보여주는 `근거 카드 클릭률`을 주요 품질 지표로 삼는다.

### 4.2 비목표
- 수만 개 에이전트를 돌리는 초대형 멀티버스 시뮬레이션
- 기업용 HR/조직궁합 B2B 상품화
- 실시간 강화학습 기반 "최적 인생 네비게이션"
- 유명인/제3자 사주 조회 플랫폼
- 서양 점성술, 타로, 심층 RAG 고서 해석의 동시 출시

## 5. 핵심 원칙

### 5.1 결정론 우선
- 출생 입력이 같으면 차트와 기본 타임라인은 언제나 같아야 한다.
- 월운/세운/대운 계산은 엔진과 정책 버전으로 재현 가능해야 한다.
- LLM은 계산 결과를 바꾸지 못한다.

### 5.2 설명 가능성 우선
- 모든 이벤트는 `cause_features`, `impact_vector`, `evidence_refs`, `confidence`를 가져야 한다.
- 유료 리포트는 "근거 없는 길흉 문장"이 아니라 `원인 + 사건 + 대안`을 제공해야 한다.

### 5.3 월 기본, 주 확대
- 무료/기본 유저는 연간 월 타임라인과 월별 상단 사건만 본다.
- 유료 유저만 특정 월을 주 단위로 확대할 수 있다.
- 일 단위 디테일은 추후 단건 상품 또는 고급 기능으로만 검토한다.

### 5.4 시뮬레이터답게 보여주기
- 단순 문단 대신 `도메인 게이지`, `연간 타임라인`, `사건 카드`, `실시간 진행 스트림`으로 결과를 보여준다.
- 프론트는 "보고서를 읽는 느낌"보다 "미래 시뮬레이션을 관찰하는 느낌"을 줘야 한다.

### 5.5 비용과 품질 동시 통제
- 차트 컨텍스트는 가능한 한 `compact 포맷`으로 고정한다.
- 이벤트 후보 생성은 규칙 기반으로 우선 처리하고, LLM은 상위 이벤트의 서술 캐시에만 사용한다.
- 확대 모드는 호출 수와 토큰 상한을 플랜 정책과 연결한다.

### 5.6 개인정보와 책임 경계
- 생년월일, 출생시, 출생지는 민감 정보에 준해 취급한다.
- 금융/의료/법률적 단정 표현을 금지하고, 확률과 경향 표현을 유지한다.
- 유명인 악용, 타인 동의 없는 입력, 악의적 공유 시도를 차단한다.

## 6. MVP 범위

### 6.1 MVP에 반드시 포함
- 이메일 또는 소셜 기반 계정 생성/로그인
- 출생 정보 입력과 계산 정책 선택
- 차트 컴파일 API와 결과 저장
- 연간 기본 시뮬레이션 생성
- 월별 상단 사건 1~3개 노출
- 재물/커리어/관계/건강 도메인 게이지
- 무료 티저 리포트
- 유료 전체 리포트와 월 확대(주 단위)
- 결제와 결제 상태 동기화
- 백테스트 입력
- 약관/개인정보 동의 및 데이터 삭제 요청

### 6.2 출시 직후 확장 후보
- 2인 궁합/동업 시뮬레이션
- 일기/감정/수면/지출 기반 현실 동기화
- 공유용 이미지 생성
- 푸시 알림
- 상담형 챗 인터페이스

### 6.3 명시적 제외
- 전문가 상담 마켓플레이스
- 사주 외 운명학 플러그인 상용화
- B2B 조직 적합도 대시보드
- 초대형 에이전트 군집 시뮬레이션

## 7. 핵심 사용자 플로우

### 7.1 출생 정보 입력과 차트 컴파일
1. 사용자가 생년월일시, 출생지, 성별, 양력/음력, 윤달 여부를 입력한다.
2. 사용자는 질문 카테고리(예: 이직/연애/투자)와 리스크 성향을 고른다.
3. 사용자는 선택적으로 현재 직업, 관계 상태, 거주 형태, 자산 대역, MBTI 같은 `보조 컨텍스트`를 입력할 수 있다.
4. 시스템은 정책 버전(`태양시`, `절기 기준`, `야자시`)을 명시한 뒤 차트를 컴파일한다.
5. 사용자는 컴파일 진행 상태를 실시간으로 본다.

수락 기준:
- 잘못된 입력 형식은 즉시 검증한다.
- 같은 입력 + 같은 정책 버전이면 항상 같은 차트가 생성된다.
- 차트 결과에는 `engine_version`, `schema_version`, `policy_version`이 포함된다.
- 동일한 생년월일시를 가진 사용자라도 `보조 컨텍스트` 차이로 시뮬레이션 서술과 중요도 우선순위가 달라질 수 있어야 한다.

### 7.2 연간 타임라인 확인
1. 사용자는 12개월 타임라인과 각 월의 `긴장도`, `기회도`, `상위 도메인`을 본다.
2. 각 월마다 상위 사건 카드 1~3개를 확인한다.
3. 무료 사용자는 티저 문장만, 유료 사용자는 전체 근거/대안을 본다.

수락 기준:
- 모든 월에는 최소 1개의 카드 또는 "평온한 구간" 상태가 존재한다.
- 동일 원인의 중복 이벤트는 클러스터링하여 하나의 대표 카드로 보인다.

### 7.3 특정 월 확대
1. 사용자가 관심 월을 누른다.
2. 시스템이 주 단위 확대 작업을 큐에 넣고 WebSocket으로 진행률을 스트리밍한다.
3. 사용자는 4~5주 타임라인과 세부 이벤트를 확인한다.

수락 기준:
- 확대는 유료 플랜에서만 가능하다.
- 확대 시작 시 사용자에게 예상 대기시간과 남은 크레딧/쿼터를 보여준다.

### 7.4 유료 리포트 해금
1. 시스템은 가장 중요한 월/사건을 `클라이맥스 카드`로 제시한다.
2. 무료 사용자는 티저와 일부 근거만 본다.
3. 결제 후 `content_paid`, `evidence_cards`, `avoidance_plan`, `대화 스크립트`가 열린다.

수락 기준:
- 페이월은 설명 가능한 이벤트에만 걸린다.
- 결제 후 동일 이벤트/리포트는 재결제 없이 재조회 가능하다.

### 7.5 백테스트
1. 사용자가 과거 연도를 선택한다.
2. 시스템이 월별 사건 후보를 보여준다.
3. 사용자는 발생/미발생/애매/기억 안 남 + 강도를 기록한다.
4. 시스템은 내부 정합성 지표를 업데이트한다.

수락 기준:
- 백테스트는 사용자에게 "정답/오답" 대신 "얼마나 맞닿았는지"를 보여준다.
- 피드백은 모델 개선에 쓸 수 있도록 구조화 저장한다.

## 8. 도메인 로직

### 8.1 계산 파이프라인
1. `입력 정규화`
2. `차트 컴파일`
3. `파생 피처 계산`
4. `월/주 타임라인 투영`
5. `이벤트 후보 생성`
6. `중요도 계산 + 클러스터링`
7. `리포트 섹션 생성`
8. `유료/무료 게이트 적용`

### 8.2 상태머신이 관리할 값
- `traits`: 결단성, 안정추구, 관계민감도, 모험성
- `resources`: 재물, 건강, 관계, 커리어 게이지
- `defense_patterns`: 회피, 돌파, 통제, 의존 같은 행동 편향
- `current_luck_modifiers`: 월운/세운에 따른 민감도 변화
- `goal_weights`: 사용자가 중요하게 보는 도메인 가중치
- `context_overlays`: 직업, 관계 상태, 자산 대역, 거주 형태, MBTI 같은 현실 컨텍스트
- `npc_roles`: 귀인, 경쟁자, 상사, 파트너 같은 환경/조연 역할
- `tradeoff_rules`: 특정 선택이 재물은 올리지만 건강/관계를 깎는 교환비 규칙

### 8.3 이벤트 중요도 기준
이벤트 노출 우선순위는 다음 값을 조합해 계산한다.

- `ImpactScore`: 4개 도메인 게이지 변화량
- `NoveltyScore`: 최근 유사 이벤트와의 중복도
- `UserFocusScore`: 사용자가 질문한 카테고리와의 일치도
- `CausalClarityScore`: 룰 기반 근거가 얼마나 선명한지
- `EmotionDeltaScore`: 현실 동기화 데이터가 붙은 경우 체감 변화량
- `ConfidenceScore`: 규칙 기반 비중

### 8.4 엣지 케이스와 보정
- `쌍둥이/동일 명식`: 동일 차트라도 보조 컨텍스트와 최근 현실 입력으로 시뮬레이션 우선순위를 조정한다.
- `출생시 미상`: 시간 미상 입력은 신뢰도 하향, 시간 후보별 분기 비교 UX를 제공할 수 있어야 한다.
- `유명인/제3자 악용`: 신고, 차단, 생성 제한 정책을 운영 도구와 연결한다.
- `트레이드오프 표시`: 모든 핵심 이벤트는 "무엇을 얻고 무엇을 잃는지"를 한 줄로 설명해야 한다.

## 9. 기술 아키텍처

### 9.1 권장 스택
- Frontend: `Next.js + TypeScript + Tailwind CSS`
- API: `FastAPI`
- Worker: `Celery`
- Database: `PostgreSQL(JSONB 적극 활용)`
- Cache/Stream: `Redis`
- Object Storage: `S3 호환 스토리지`
- Saju Engine: `ssaju(TS)` 또는 `sajupy(Python)` 중 하나를 핀 고정
- LLM: `OpenAI-compatible API`

### 9.2 아키텍처 원칙
- API 서버는 검증, 인증, enqueue, 스트리밍 허브 역할만 담당한다.
- 장시간 작업은 모두 워커에서 처리한다.
- 엔진 출력은 버전과 함께 저장한다.
- 원본 로그와 사용자용 요약은 분리 저장한다.
- 이벤트는 append-only로 저장하고, 수정 대신 보정 이벤트나 재생성 버전을 남긴다.

### 9.3 실시간 UX
- 차트 컴파일, 확대 시뮬레이션, 리포트 생성은 WebSocket으로 진행 상황을 푸시한다.
- 사용자에게는 `현재 단계`, `최근 생성 사건`, `예상 대기시간`을 보여준다.

## 12. 법률, 보안, 윤리

### 12.1 라이선스
- MiroFish는 AGPL 리스크가 있으므로 구현 참고 소스가 아니라 요구사항 참고 소스로만 취급한다.
- ADR에 `클린룸 의사결정`을 남기고, 구현 문서에는 MiroFish 표현을 복제하지 않는다.

### 12.2 개인정보
- 출생 정보는 암호화 저장한다.
- 사용자는 데이터 열람, 내보내기, 삭제를 요청할 수 있어야 한다.
- 일기/감정/지출 데이터는 별도 보존 기간을 두고 더 짧게 관리한다.

### 12.3 표현 가이드
- "반드시", "무조건", "확정" 같은 표현 금지
- 의료/재무/법률 의사결정은 참고 정보임을 명시
- 해석은 `확률`, `경향`, `주의`, `권장 행동` 중심

### 12.4 악용 방지
- 제3자 생년월일시 입력에 대한 신고/차단 정책
- 유명인/공인 대상 악성 생성 방지
- 공유 이미지에 개인정보 최소 노출

## 14. 최종 제품 판단 기준
이 프로젝트는 다음 4개가 동시에 성립할 때 "올바르게 만들어졌다"고 판단한다.

- 사용자가 텍스트 운세가 아니라 `움직이는 시뮬레이션`으로 인식한다.
- 이벤트 하나하나가 `설명 가능`하다.
- 결제가 단순 차단이 아니라 `행동 지침을 여는 순간`처럼 느껴진다.
- 동일 입력에 대해 결과가 재현 가능하고, 운영 비용이 통제된다.
